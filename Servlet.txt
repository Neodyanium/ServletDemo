* Client and Server Architecture

- Any device/computer where we can use internet to access via browser to access some service then it is a client.
- The client is a user or system that is request a specific resource or service from an entity called as Server 

-Internet is a global communication system that links thousands of individual network. It allows the exchange of info between 2 or more computers on a network.

- A machine where you make your service available to people globally can be called as a Server
- A sever maybe responsible to process one request or more on request time

- Webserver is a software program or server computer which has a world wide web access 


- The interaction between the webserver and webbrowser is done with the http protocol
- When we want to access the service we need to provide the web browser the address(IP) of the machine and if the server has multiple applications we need to provide the port address, but all this things are handled by the DNS (domain name server, ig it is like a registry where they map server name to the server ip)



* Static and Dynamic Response

- Static response is when the output doesn;t change with User. (Html, CSS)

- Dynamic response is which will change with the user (Servlet, JSP, php etc)

consider an eg, You are filling a registration form so that is a static html/css page, whereas upon registration you receive, Your Name + Registration details that is a dynamic response

We use Spring MVC framework, which is build on top of the tomcat web server, it is abstracted out.


*Servlet

-Whenever we want our Server to provide a dynamic response we will use a Servlet.

- In a webserver we will have a server container, that is a runtime environment that is reponsible to run the java code. It does not have the main method, web container/servlet container runs it no jvm involved. 

-To make it a server you will require a server software like Apache Tomcat

- Servlet is a Java program, residing on the server computer it is responsible to process the incoming http service and provide the dynamic response back to client. This can include complex logic and database connnections. A class which implements a Servlet or extends HttpServlet class will be a Servlet.

* To develop a Servlet application we require Java EE perspective in Eclipse IDE, not sure about others yet and a server application like Apache Tomcat, we can configure the server from IDE, we should set non busy port for tomcat, there is a separate port for Admin

* Lifecycle of Servlet

- The webcontainer is responsible for running the servlet. Many methods comes when we implement HttpServlet and they are called internally.

- The Servlet class works like any other when it comes to class loading, so any static blocks will be executed, the comes the class instantiation so a constructor will run, after this the init method is called internally(any initialisation tasks can be done under these blocks), init used for db connections or configs reading 

-After this the service method is called when we receive a request, for each request a new thread is created and each thread then runs the service method. Service creates a request and response object, Service directs the request to doPost() or doGet() I feel it depends on the request method type

- At the end, when shutting down Tomcat, container calls a destroy method is called which shuts down the servlet

- The init() and destroy() methods are only called once in a Lifecycle, only Service can be called multiple times

Qs, The responsibiliity to create a Servlet is of the Web Container or Servlet container (here it is Tomcat). It works lazily, so instanciation of a servlet happens when a request is received and not when we start the server.

** doPost and doGet methods are designed to handle the specific requests, the routing is done by service method. It is a good practice to handle static pages via the get method, like looking for some kind of form. Whereas it is good to input data via a post method. 

** Overriding a service method can handle both the post and get requests. Only issue is if you have get and post methods serving different purpose then needs if and else block to check what kind of request it is then providing the response. 

** I have personally cheked, in a post type html form if I used a service method or doPost method, after sending data and getting a response, reloading the page asks if you want to resubmit and it doesn't get to the input page but resubmits the current data again, so you will have duplicate data. This happens with get type forms too only the warning don't pop up. 

*****The PRG (Post/Redirect/Get) Pattern, Under this the doPost() and doGet() has simple responsibilities, doPost() recieves and process data, add it to database or any other thing it does not render HTML. Instead, it ends by executing a redirect, to a view(get method). doGet() Its sole job is to fetch and display data It simply reads the current state and uses PrintWriter or a JSP file to display the final result, doesn't change any data. This is how multiple reload will have no effect, since at the end we are getting a view, a get request that doesnt change the data.
